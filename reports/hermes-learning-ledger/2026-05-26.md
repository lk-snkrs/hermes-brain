## Aprendizado 1 — SPITI provider fix precisa virar monitoramento, não expansão automática

- Origem: sessão recente de Lucas sobre `SPITI HERMES` com falha `Provider authentication failed: No Codex credentials stored`.
- Classificação: execução/runtime profile; A1/A2 para diagnóstico e config confinada ao profile, A3 para mudanças de persistência/restart amplo.
- Destino aplicado: relatório diário 02h e prioridade ativa de monitoramento SPITI.
- Ação feita: registrada a correção como contexto operacional e mantido o modo atual como monitoramento, sem reiniciar Docker/gateway global.
- Verificação: cron mostra `SPITI Telegram gateway watchdog` ativo e último run OK; helper host sem alertas.
- Bloqueio: voltar SPITI para Codex exige reautenticação OAuth específica; restart/gestão persistente de gateway exige plano/rollback se afetar runtime.
- Próxima regra: provider fallback em profile especialista deve ser isolado, verificado por teste e acompanhado por watchdog antes de virar mudança estrutural.

- Classificação: runtime reconciliation; A0/A1 read-only.
- Destino aplicado: relatório diário 02h.
- Ação feita: verificados jobs críticos por nome antes de classificar drift.
- Verificação: Daily loop, Fechamento 23h, relatório 02h30, Mesa COO, SPITI/LK Growth/Mordomo watchdogs encontrados; host helper com `alerts: []`.
